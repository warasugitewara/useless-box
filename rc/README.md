## 📌 位置づけ

本体（意味のない箱）に**統合しない**独立系統です。理由は電気的な分離にあります。

ブラシ付きDCモーターは強いノイズ源で、電源やGNDを共有するとサーボの誤動作や
DFPlayer 導入後のスピーカーノイズの原因になり得ます。**別系統にすれば問題ごと消えるため**、
本体は2電源構成のまま維持し、走行機構は完全に切り離しました。

その結果、本体側では制御ピンを一切消費せず `D7`〜`D9`・`D13` が空いたままになっています。

## 🧱 構成

```
[送信機]  ESP32 + ジョイスティック
              │
              │  ESP-NOW（ルーター不要・直接通信）
              ▼
[車体]    ESP32 + TA7291P ×2 + クローラー
```

| 項目 | 内容 |
|:-----|:-----|
| マイコン | Freenove ESP32 開発ボード（2個パック・カメラなし） |
| 通信 | **ESP-NOW** — ルーター不要、遅延数ms |
| 操作部 | ジョイスティックモジュールを自作コントローラ化 |
| モータードライバ | **TOSHIBA TA7291P** ×2 |

## ⚠️ 設計上の注意点

実装前に洗い出した、踏みそうな地雷の一覧です。

### ESP32 側

| # | 内容 |
|:-:|:-----|
| 1 | **ADC2 は WiFi 有効時に読めない。** ESP-NOW は WiFi 無線を使うため、ジョイスティックのアナログ2軸は必ず **ADC1（GPIO 32/33/34/35/36/39）** に接続する |
| 2 | **ジョイスティックの電源は 3.3V から取る。** 5V だとアナログ出力が ADC の入力上限（3.3V）を超える |
| 3 | GPIO 34〜39 は**入力専用でプルアップが無い**。押しボタンは GPIO 32/33/27 など、プルアップが効くピンへ |
| 4 | GPIO 0/2/5/12/15 は**ストラッピングピン**、6〜11 は**フラッシュ用**。いずれも避ける |
| 5 | ESP32 Arduino core **3.x で LEDC(PWM) の API が変更済み**。`ledcSetup()`/`ledcAttachPin()` を使う古い記事のコードはそのままでは通らない |
| 6 | WiFi 送信時に瞬間的な電流スパイクが出る。**モーター電源と ESP32 電源は分離する** |

### TA7291P 側

| # | 内容 |
|:-:|:-----|
| 7 | **`Vs`（モーター電源）は `Vref` より高い必要がある。** Vref に 3.3V PWM を入れるなら Vs は余裕をもって 6V 以上。バイポーラ出力段ぶんの電圧降下もある |
| 8 | **1個で1モーター。** 左右2モーターには **2個必要** |
| 9 | Vcc（ロジック電源）は 3.3V で動作するため、**ESP32 との間にレベル変換は不要** |
| 10 | データシートに「逆起電力によりモータ側電源端子・出力端子が定格電圧を超えないように設計すること」と明記あり。**モーター端子間に 0.1µF セラミックコンデンサ**を入れて緩和する |
| 11 | クローラーは障害物で簡単にストールし、その瞬間に定格を超える電流が流れる。モーターのストール電流とICの出力電流定格を突き合わせておく |
| 12 | **新規製造中止の方向**。手持ちは2個のみで予備が無いため、破損＝プロジェクト停止。予備の確保を検討する |

### 通信

| # | 内容 |
|:-:|:-----|
| 13 | **フェイルセーフを必ず実装する。** 受信側で最後のパケットから一定時間（500ms 程度）経過したらモーターを停止させる。ESP-NOW は「届かなくても何も起きない」だけなので、これが無いと通信途絶時に暴走し続ける |
| 14 | 送受信で**同じ WiFi チャンネル**にすること |
| 15 | 最初はブロードキャストアドレス `FF:FF:FF:FF:FF:FF` で組むと MAC アドレス調べが不要 |
| 16 | ESP32 の ADC は 12bit（0〜4095）。ジョイスティックの中央値には個体差があるため、**起動時にセンター値をキャリブレーション**し ±150 程度のデッドゾーンを設ける |

## 📂 ファイル構成（予定）

```
rc/
├── README.md          # このファイル
└── crawler_rc.ino     # 送信機・受信機 兼用（未実装）
```

送信機と受信機は `#define ROLE` で切り替える**1ファイル構成**にします。

ESP-NOW は送受信でパケット構造体が 1バイトでもズレると壊れるため、
ファイルを分けると「片方だけ修正して事故る」典型的な失敗が起きます。
**構造体の定義が1箇所にしか存在しない**状態を作れることを優先しました。

書き込む際は `ROLE` を書き換えてから、それぞれのボードへアップロードします。
